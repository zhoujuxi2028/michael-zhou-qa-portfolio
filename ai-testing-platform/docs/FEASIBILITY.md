# AI Testing Platform — 可行性分析

## 执行摘要

**结论：技术可行，建议实施。**  
利用规则引擎 + 统计分析可在不依赖外部 LLM API 的情况下展示 AI 驱动测试的核心价值，适合 Portfolio 展示。

---

## 1. 背景与问题定义

### 现状

当前 QA 测试流程的主要痛点：

| 痛点 | 影响 |
|------|------|
| 测试用例手工编写效率低 | 每个 Sprint 需要数小时手工梳理测试场景 |
| 缺陷发现滞后 | 高风险模块缺乏提前预警机制 |
| 测试脚本模板重复劳动 | AAA 框架、fixture 配置反复手写 |

### 目标

探索 AI 技术在测试流程三个关键环节的应用：
1. **需求 → 测试用例**：自动提取测试场景
2. **代码度量 → 风险预测**：提前锁定高危模块
3. **测试规范 → 测试脚本**：消除脚手架重复工作

---

## 2. 技术方案对比

### 方案一：集成外部 LLM（GPT-4/Claude）

| 项目 | 评估 |
|------|------|
| 实现质量 | ⭐⭐⭐⭐⭐ 高质量自然语言理解 |
| 成本 | ❌ API 费用高，不适合 Portfolio CI 环境 |
| 依赖 | ❌ 需要网络连接和 API Key |
| 测试难度 | ❌ 难以在 CI 中稳定测试（Token 限制、速率限制） |
| 适合度 | ⚠️ 生产级产品合适，Portfolio 展示过重 |

### 方案二：规则引擎 + 统计模型（本项目选择）✅

| 项目 | 评估 |
|------|------|
| 实现质量 | ⭐⭐⭐ 逻辑清晰，覆盖主流场景 |
| 成本 | ✅ 零成本，无外部依赖 |
| 依赖 | ✅ 纯 Python 标准库 |
| 测试难度 | ✅ 完全可以编写单元测试，行为确定 |
| 适合度 | ✅ Portfolio 展示最优选择 |
| 扩展性 | ✅ 接口设计支持未来替换为 LLM 后端 |

### 方案三：本地轻量级 NLP（spaCy/transformers）

| 项目 | 评估 |
|------|------|
| 实现质量 | ⭐⭐⭐⭐ NER + 关系提取 |
| 成本 | ⚠️ 模型下载（>100MB），CI 构建慢 |
| 依赖 | ⚠️ spaCy/transformers 安装复杂 |
| 测试难度 | ⚠️ ML 模型输出非确定性，测试不稳定 |
| 适合度 | ⚠️ 可作为后续升级方向 |

**选择方案二**，理由：零依赖、完全可测试、CI 友好、展示核心思路最直接。

## 3. 技术可行性评估

### 3.1 智能测试用例生成

**技术实现**：关键词提取 + 场景模板映射

```
需求文本 → 词法分析 → 关键词识别 → 场景模板匹配 → 优先级推断 → TestCase 列表
```

| 评估维度 | 结论 |
|---------|------|
| 技术难度 | 低 — 正则 + 字典匹配 |
| 覆盖率 | 中 — 覆盖 CRUD、安全、边界等主流模式 |
| 限制 | 复杂业务逻辑需人工补充 |
| 可行性 | ✅ 完全可行 |

**有效场景**（关键词覆盖）：
- CRUD 操作：`create`, `read`, `update`, `delete`, `login`, `search`, `upload`, `validate`
- 安全场景：`authentication`, `authorization`, `password`, `token`, `input`, `query`, `file`, `api`
- 边界条件：数值模式 `\d+ (characters|items|MB|GB)`，`max/min/limit` 关键词

### 3.2 缺陷预测

**技术实现**：加权风险评分模型

```
ModuleMetrics → 因素归一化(0-100) → 加权求和 → 风险等级分类 → 建议生成
```

| 风险因素 | 权重 | 依据 |
|---------|------|------|
| 圈复杂度 | 25% | IEEE/SEI 研究：CC>10 缺陷率 × 2 |
| 代码变更频率 | 25% | Google 研究：高频变更模块缺陷率高 |
| 覆盖率缺口 | 20% | 低覆盖模块未发现缺陷概率高 |
| 历史缺陷数 | 20% | 缺陷聚集效应（Pareto 80/20）|
| 代码规模 | 10% | 大模块维护难度高 |

| 评估维度 | 结论 |
|---------|------|
| 技术难度 | 低 — 纯数学计算 |
| 精度 | 中 — 基于启发式规则，实际精度取决于权重校准 |
| 限制 | 无 ML 训练数据需求 |
| 可行性 | ✅ 完全可行 |

### 3.3 自动化脚本生成

**技术实现**：模板引擎 + 结构化填充

```
TestSpec → 模板选择 → 代码块生成(Arrange/Act/Assert) → 脚本组装 → 质量验证
```

| 评估维度 | 结论 |
|---------|------|
| 技术难度 | 低 — 字符串模板 + 格式化 |
| 输出质量 | 高 — 结构化、可直接运行 |
| 限制 | 生成的脚本需要人工填充业务逻辑细节 |
| 可行性 | ✅ 完全可行 |

## 4. 风险评估

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| 需求文本理解不准确 | 中 | 中 | 明确支持的关键词范围；提供示例输入 |
| 风险权重设置不合理 | 低 | 中 | 权重可配置化，未来基于真实数据校准 |
| 生成脚本语法错误 | 低 | 高 | validate_generated_script 验证 + 单元测试全覆盖 |
| CI 中断 | 低 | 低 | 零外部依赖，CI 稳定 |
