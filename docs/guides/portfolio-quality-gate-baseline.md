# Portfolio 统一质量门禁基线（DoD Baseline）

> 目标：先补治理短板，再推进平台化能力。  
> 适用范围：仓库级 + 全部子项目（Node.js / Python / 性能 / 安全 / K8S）。

---

## 1. 最小门禁（必须满足）

| 类别 | 最小门禁 | 说明 |
|------|----------|------|
| Lint | 项目既有 lint 全绿 | 不新增临时 `|| true` 掩盖 |
| Test | 单测必须通过；有集成/E2E 的项目需通过对应测试 | Stage 4 不补写新测试 |
| Coverage | 增量覆盖率不下降（相对基线） | 允许存量债务，禁止继续下滑 |
| Security | 依赖与代码扫描必须有结果，P0/P1 不得忽略 | 与缺陷跟踪系统联动闭环 |

---

## 2. 覆盖率增量不下降规则

| 规则 | 执行要求 |
|------|----------|
| 基线定义 | 使用目标分支最近一次绿色构建覆盖率作为 baseline |
| 判定方式 | PR 覆盖率 < baseline 时标记为 failed |
| 例外机制 | 必须登记 Defect/Waiver，写明到期修复时间 |
| 责任归属 | 代码作者提交证据，Reviewer 负责 gate 签核 |

---

## 3. Commit Subject 预检（开发前置）

> 规则：`subject ≤ 72`，且符合 Conventional Commits。  
> Cloud Agent 通过 `report_progress` 提交会绕过 Husky，必须做本地脚本预检。

```bash
bash scripts/check-commit-guard.sh
```

---

## 4. 可观测性统一指标（10 项目共用）

| 指标 | 定义 |
|------|------|
| 通过率 | pass / total |
| 失败 Top | 按失败次数排序的测试/作业 |
| 平均耗时 | 测试作业平均执行时间 |
| Flaky 率 | 同用例在固定窗口内 fail/pass 波动比例 |
| 失败分类占比 | 代码缺陷 / 环境波动 / 数据问题 / 依赖问题 |

Nightly/Soak/Security 不只看单次绿灯，必须看趋势。

---

## 5. Flaky 治理标准

| 分级 | 定义 | 处理 |
|------|------|------|
| P0 | 阻塞主线发布或核心回归 | 立即修复，不允许重试掩盖 |
| P1 | 高频波动（影响评审结论） | 进入优先修复队列 |
| P2 | 低频波动（可观测） | 进入治理 backlog，按周期收敛 |

要求：
- 记录“重试后通过”与“根因修复后通过”为不同状态
- 连续 N 次不稳定必须自动标记并指派 owner（默认 N=3）

---

## 6. 安全左移（供应链）

| 项 | 最小要求 |
|----|----------|
| SBOM | 各核心项目按周期生成与归档 |
| 依赖巡检 | 按项目周期检查高危依赖 |
| SLA | 高风险依赖定义限时修复（见 `dependency-risk-sla.md`） |
| 闭环 | 扫描结果必须落入 defect tracking，不做孤立报告 |

---

## 7. 测试数据与环境一致性

- 测试数据统一分层：固定基线 / 场景数据 / 压测数据
- 本地、CI、夜间任务使用一致的关键依赖版本与启动顺序
- 端口、容器、外部依赖冲突必须可快速定位（命令 + 责任人）

详见：[`test-data-environment-standard.md`](test-data-environment-standard.md)

---

## 8. 文档防漂移机制

- workflow 新增或改名时，必须同步 `README.md` 与 `CLAUDE.md`
- PR 级自动校验：`scripts/check-workflow-doc-sync.sh`
- 发生缺陷后，RCA 需沉淀“预防动作”字段并回写流程文档

---

## 9. 组合测试策略（价值提升）

核心项目优先落地：
- 契约测试 + 变更影响分析（减少全量回归）
- 风险驱动测试优先级（高风险模块先测）
- 业务关键路径“冒烟最小集”固化（缩短反馈回路）
