## ⚠️ 核心风险

### 1. 🔴 Python 环境（选择风险）

**问题:** 项目是 Node.js，用 Python 脚本显得突兀

| 影响 | 说明 |
|------|------|
| 依赖链 | 需要 Python 3.8+ + pip + 依赖包 |
| CI 兼容性 | performance-ci.yml 需要新增 Python setup step |
| 可维护性 | 团队主要用 Node.js，Python 脚本难维护 |
| 建议 | 改用 Node.js 可能更合适 |

**替代方案:** Node.js 脚本？(易维护、与项目一致)

---

### 2. 🔴 Docker 强制启动（选项 A 风险）

**问题:** "强制要求 Docker/OrbStack 运行，未启动则报错"

| 影响 | 说明 |
|------|------|
| 开发阶段 | 开发者可能未启动 OrbStack，脚本就报错 |
| CI 环境 | GitHub Actions 需要 Docker available |
| 失败率 | OrbStack 偶尔不稳定，导致脚本失败 |
| 用户体验 | 强制失败不如智能 SKIP 友好 |

**建议:** 改为选项 B（检查状态，未启动则 SKIP）？

---

### 3. 🟡 手工验证（启动服务）难度高

**问题:** 脚本需要启动 npm start，运行测试，再停止

| 挑战 | 风险 | 解决方案 |
|------|------|---------|
| 端口冲突 | npm start 可能端口被占用 | 使用 lsof 检查，或绑定动态端口 |
| 竞态条件 | curl 在 npm start 完成前运行 | 添加 health check + sleep |
| 信号处理 | npm stop 可能无法优雅关闭 | 使用 SIGTERM → SIGKILL 梯级关闭 |
| Rate Limiter | RATE_LIMIT_ENABLED 环境变量传递 | 确保 npm start 正确继承环境 |
| k6 smoke 输出解析 | k6 输出格式复杂 | 正则表达式匹配 p95、error rate |

**可行性:** 75% — 需要仔细处理 3-4 个陷阱

---

### 4. 🟡 gh CLI 自动安装

**问题:** 脚本自动安装 gh 可能失败

| 场景 | 风险 |
|------|------|
| CI 容器中 | apt-get / brew 权限不足 |
| 网络问题 | 下载失败 |
| 已安装冲突版本 | 版本不兼容 |

**可行性:** 85% — 可以处理，但需要 fallback

---

### 5. 🟡 超时设置（5 分钟）偏短

| 检查项 | 实际耗时 | 超时 | 风险 |
|--------|---------|------|------|
| npm test | ~55 秒 | 5 分钟 | ✅ 安全 |
| 集成测试 | ~3-5 分钟 | 5 分钟 | 🟡 紧 |
| k6 smoke | ~2-3 分钟 | 5 分钟 | ✅ 安全 |

**建议:** 集成测试超时改为 8 分钟？

---

### 6. 🟡 配置管理（同时支持分离 + 硬编码）

**问题:** 用户同时勾选两个看似矛盾的选项

**解释:** 可能是想要两种方式都支持
- 快速开发：硬编码
- 长期维护：配置文件

**实现复杂度:** 中等 — 需要抽象出配置层

## 🛠️ 建议调整

### 强烈建议

1. **改用 Node.js** 替代 Python
   - 原因：项目生态一致、团队技能、维护性
   - 工作量影响：-0.5 小时（Node.js 实际简化）

2. **Docker/OrbStack 改为选项 B**（检查状态，未启动则 SKIP）
   - 原因：更友好的用户体验
   - 工作量影响：+0.5 小时（但值得）

3. **集成测试超时改为 8 分钟**
   - 原因：当前 5 分钟太紧，易超时失败

### 可选建议

4. **简化配置管理**
   - 目前硬编码即可，Phase 7 再分离
   - 工作量节省：-1.5 小时

5. **k6 smoke 输出解析**
   - 先正则表达式匹配，失败时显示完整输出给用户
   - 避免脚本过度智能导致难以调试
