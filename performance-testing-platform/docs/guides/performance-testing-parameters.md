# 性能测试参数设定指南

---

## 1. 四种测试级别的本质区别

不是"并发数从小到大跑四遍"，每个级别回答的问题不同：

| 级别     | 回答的问题            | 失败意味着什么            |
| ------ | ---------------- | ------------------ |
| smoke  | 系统能不能跑起来？        | 部署失败、配置错误、依赖不可用    |
| load   | 正常负载下性能是否达标？     | SLA 不达标，需要优化或扩容    |
| stress | 系统的极限在哪里？超过后会怎样？ | 不知道容量上限，无法做容量规划    |
| spike  | 突发流量后能否自动恢复？     | 促销/突发事件后系统挂死，需人工重启 |
|        |                  |                    |

### 为什么是这四种，不是三种或五种？

这四种覆盖了性能测试的两个维度：

```
            负载稳定性
            ↑
  load ─────┤──────── stress
  (稳态验证) │         (极限探索)
            │
  smoke ────┤──────── spike
  (基线确认) │         (瞬态恢复)
            └──────────────────→ 负载强度
```

- **纵轴（稳定性）：** smoke/spike 是短时验证，load/stress 是持续观察
- **横轴（强度）：** smoke/load 在预期范围内，stress/spike 超出预期

## 2. Little's Law — 性能测试的第一原理

### 基础公式

```
L = λ × R

L = 系统中正在处理的平均请求数（in-flight requests）
λ = 吞吐量（requests/sec）
R = 平均响应时间（seconds）
```

> **注意：L ≠ threads。** L 是任意时刻服务器正在处理的请求数，threads 是虚拟用户总数（包含处于 think time 的用户）。

### 闭合系统扩展（JMeter / k6）

性能测试工具使用固定数量的虚拟用户循环发请求，属于**闭合系统**。由 Little's Law 推导出：

```
N = λ × (R + Z)

N = 虚拟用户数（JMeter threads / k6 VUs）
λ = 吞吐量（requests/sec）
R = 平均响应时间（seconds）
Z = Think time（seconds）
```

两个公式的关系：

| 公式 | 视角 | 回答的问题 |
|------|------|-----------|
| L = λ × R | 服务器端 | 任意时刻有多少请求正在处理中？ |
| N = λ × (R + Z) | 用户端 | 需要多少虚拟用户才能产生目标吞吐量？ |

当 Z = 0 时，N = L（每个用户在响应返回后立即发下一个请求）。

### 实际应用

**场景 1: 已知业务需求，推算 threads**

```
需求: 峰值 100 req/s，预期响应时间 200ms

无 think time:   N = 100 × 0.2       = 20 threads
加 think time 1s: N = 100 × (0.2 + 1.0) = 120 threads
```

> think time 从 0→1s，线程数从 20→120（6 倍）。
> 这解释了为什么 think time 是参数设计中最敏感的变量。

**场景 2: 已知 threads，推算理论吞吐量**

```
本项目 JMeter load test: 50 threads, think time 1000ms/请求, avg response 17ms

λ = N / (R + Z) = 50 / (0.017 + 1.0) = 49.2 req/s（理论上限，50 线程全部满载）
```

> **理论 vs 实际差异的常见原因：**
> - ramp-up 阶段线程逐步启动，平均活跃线程 < N
> - 负载上升后 R 增大，降低实际 λ
> - 测试工具自身的定时器精度、采样器切换开销

**场景 3: 反向验证测试结果**

```
JMeter stress test: peak 200 threads (4 阶段 ramp-up), think time 500ms/请求, avg response 26ms

服务器视角 — 活跃请求数 (L = λ × R):
  L = 138.4 × 0.026 = 3.6
  → 200 线程中仅 3.6 个请求在处理中，其余在 think time
  → 合理：Z (500ms) >> R (26ms)

用户视角 — 有效并发验证 (N = λ × (R + Z)):
  N_effective = 138.4 × (0.026 + 0.5) = 72.8
  → 全程平均等效于 ~73 个满载线程
  → 因为 4 阶段 staged ramp-up (50→100→150→200)，
    200 线程仅在测试后半段达到，整体平均远低于 200
  → 与 Little's Law 吻合
```

## 3. Duration（测试时长）

### 设定依据

| 级别   | 建议时长     | 理由                                  |
| ------ | ------------ | ------------------------------------- |
| smoke  | 30-60s       | 快速反馈，CI 门禁友好                 |
| load   | 满载 ≥ 5-10m | 足够暴露内存泄漏、连接池耗尽、GC 停顿 |
| stress | 3-5m         | 足够观察降级拐点和错误率攀升          |
| spike  | 1-2m         | 含突增 + 恢复观察                     |

### 为什么 load test 需要 ≥ 5 分钟？

短时间测试无法暴露的问题：

| 时间范围  | 可能出现的问题           |
| ----- | ----------------- |
| 0-60s | 连接建立开销、冷启动        |
| 1-3m  | 连接池耗尽、线程池饱和       |
| 3-5m  | 内存泄漏初现、GC Full GC |
| 5-10m | 慢查询累积、日志磁盘写满      |
| 10m+  | 内存泄漏确认、长连接超时      |

> HTML 报告时间戳精度为分钟级，duration 应 ≥ 60s 以保证报告可读。

## 4. Ramp-up（线程启动时间）

**不是经验值，必须计算。**

### 计算约束

```
约束 1: ramp-up ≤ duration × 25%    （保证满载时间足够）
约束 2: ramp-up ≥ threads × 单次迭代时间  （避免瞬间过载掩盖真实问题）
约束 3: 符合测试级别意图              （spike 故意短，load 应平缓）
```

### 约束背后的原因

| 约束   | 违反后果                              | 实际案例                                         |
| ------ | ------------------------------------- | ------------------------------------------------ |
| 约束 1 | 满载时间不够，平均指标被 ramp-up 拉低 | 本项目 #51: load P1 ramp-up 占 50%，吞吐量被拉低 |
| 约束 2 | 所有线程同时发请求，第一批全部超时    | 看起来像 spike，无法评估稳态性能                 |
| 约束 3 | load test 像 spike，spike 像 load     | 四种测试级别失去区分度                           |

### 计算示例

```
load test:
  duration = 300s, threads = 50, 单次迭代 ≈ 3s
  约束 1: ramp-up ≤ 300 × 0.25 = 75s
  约束 2: ramp-up ≥ 50 × 3 = 150s (太大 → 必须分阶段)
  → 拆成多 ThreadGroup，每阶段独立计算

spike test:
  duration = 90s, threads = 100
  约束 3: 故意快速拉升 → ramp-up = 5s（违反约束 2 是有意的）
```

### 不合理的信号

| 信号                         | 后果                           |
| ---------------------------- | ------------------------------ |
| ramp-up 占比 > 30%           | 平均指标被拉低，不代表真实负载 |
| 所有线程同时启动 (ramp-up=0) | 像 spike 不像 load             |
| 满载时间 < 总时长的 50%      | 统计意义不足                   |

## 5. Threads（并发线程数）

### 从业务需求推算

**不要拍脑袋决定 threads 数。** 正确流程：

```
1. 收集业务数据
   - 日活用户 (DAU): 10,000
   - 峰值倍率: 3x
   - 平均会话时长: 5 min
   - 每会话请求数: 20

2. 计算峰值 TPS
   峰值并发用户 = DAU × 峰值倍率 × (会话时长/天) = 10000 × 3 × (300/86400) ≈ 104
   峰值 TPS = 104 × (20/300) ≈ 6.9 req/s

3. 用 Little's Law 推算 threads
   假设 avg response = 200ms, think time = 1000ms
   threads = TPS × (response + think) = 6.9 × 1.2 ≈ 8.3 → 取 10

4. 各级别倍率
   smoke:  1x = 10 threads (取最小 2-5)
   load:   1x = 10 threads
   stress: 4x = 40 threads
   spike:  10x = 100 threads
```

### 本项目（demo 项目）

没有真实业务数据，threads 基于展示目的设定：

| 级别   | threads | 选择理由       |
| ------ | ------- | -------------- |
| smoke  | 5       | 最小验证       |
| load   | 20→50   | 展示分阶段递增 |
| stress | 50→200  | 展示阶梯式压力 |
| spike  | 5→100→5 | 展示突增和恢复 |

## 6. Think Time（思考时间）

### 本质

Think time 模拟真实用户在请求间的等待（阅读页面、填写表单）。它决定了**并发数和吞吐量的关系**。

### 公式

```
理论 TPS = threads / (think_time + avg_response_time)
```

| think time | 50 threads, 20ms response | 含义                   |
| ---------- | ------------------------- | ---------------------- |
| 0ms        | 2500 TPS                  | 纯压力测试，不模拟用户 |
| 200ms      | 227 TPS                   | API 间最小间隔         |
| 1000ms     | 49 TPS                    | 模拟用户浏览           |
| 3000ms     | 16.6 TPS                  | 模拟用户阅读页面       |

### 选择依据

| 场景                   | 建议值      | 理由                           |
| ---------------------- | ----------- | ------------------------------ |
| API 压力测试（找极限） | 0-200ms     | 最大化压力，不关心真实用户行为 |
| 模拟真实用户           | 1000-3000ms | 反映实际使用模式               |
| 页面浏览场景           | 3000-5000ms | 用户阅读内容的时间             |

### 常见误区

| 误区             | 问题                         | 正确做法                           |
| ---------------- | ---------------------------- | ---------------------------------- |
| 不加 think time  | 吞吐量虚高，不反映真实场景   | 根据场景选择合适值                 |
| think time 太长  | 需要极多线程才能产生目标 TPS | 用 Little's Law 验算               |
| 固定值 vs 随机值 | 固定值产生周期性波动         | 生产级测试用 Gaussian Random Timer |

## 7. 阈值验证

### Steady State 原则

**阈值应基于满载期间（steady state），不含 ramp-up / ramp-down 阶段。**

```
|-- ramp-up --|-------- steady state --------|-- ramp-down --|
               ↑                              ↑
               阈值验证开始                     阈值验证结束
```

| 指标             | 验证时机 | 理由                             |
| ---------------- | -------- | -------------------------------- |
| 吞吐量 (TPS)     | 满载期间 | ramp-up 期间线程未满载，TPS 偏低 |
| p95/p99 响应时间 | 满载期间 | ramp-up 首批请求含连接建立开销   |
| 错误率           | 全程     | ramp-up 期间的错误也有诊断价值   |

### 百分位数的含义

| 指标         | 含义                   | 适用场景                   |
| ------------ | ---------------------- | -------------------------- |
| avg          | 平均值，易被极端值拉高 | 粗略判断，不适合 SLA       |
| median (p50) | 50% 请求低于此值       | 反映"典型"用户体验         |
| p90          | 90% 请求低于此值       | 多数用户体验               |
| **p95**      | 95% 请求低于此值       | **业界最常用 SLA 指标**    |
| p99          | 99% 请求低于此值       | 尾部延迟，对高可用系统重要 |

**为什么选 p95 而不是 avg？**

```
示例: 100 个请求
- 95 个: 10ms
- 5 个: 1000ms
- avg = 59.5ms（看起来还行）
- p95 = 10ms（95% 用户体验很好）
- p99 = 1000ms（1% 用户体验极差）

avg 掩盖了尾部问题。p95 + p99 组合才能完整评估。
```

## 8. 结果分析 — 从数据到结论

### 如何判断测试结果是否正常

```
步骤 1: 用 Little's Law 验算
  实际 TPS × avg response ≈ 活跃线程数？
  差距大 → 有排队或阻塞

步骤 2: 看趋势而不是单点
  响应时间随时间递增 → 内存泄漏 / 连接池耗尽
  响应时间稳定 → 系统在可承受范围

步骤 3: 看分布而不是平均
  p95 和 avg 差距大 → 有长尾请求，排查慢查询
  p95 ≈ avg → 响应时间均匀，系统健康
```

### 关键指标之间的关系

```
                    ┌─ TPS 下降
并发增加 → 响应时间上升 ─┤
                    └─ 错误率上升（到达极限后）

这三个变化的拐点就是系统容量上限。
```

### 瓶颈定位

| 现象                   | 可能原因                        | 排查方向              |
| ---------------------- | ------------------------------- | --------------------- |
| 响应时间高，CPU 低     | I/O 阻塞（DB 慢查询、外部调用） | DB slow log, 网络延迟 |
| 响应时间高，CPU 高     | 计算密集（加密、序列化、GC）    | profiler, GC log      |
| TPS 上不去，资源都不满 | 线程池/连接池满、锁竞争         | 线程 dump, 连接池监控 |
| 错误率突增             | 超时、拒绝连接、OOM             | 应用日志, 系统 dmesg  |
| 响应时间随时间递增     | 内存泄漏                        | heap dump 对比        |

## 9. 容量规划（Capacity Planning）

### 9.1 什么是容量规划

容量规划回答一个核心问题：**系统需要多少资源才能支撑预期的业务增长？**

它不是一次性活动，而是一个持续循环：

```
业务预测 → 容量建模 → 压测验证 → 扩容决策 → 监控反馈
    ↑                                          │
    └──────────────────────────────────────────┘
```

### 9.2 容量规划三步法

**Step 1: 业务建模 — 从业务指标到技术指标**

```
业务指标                    技术指标
──────────                  ──────────
日活用户 (DAU)         →    峰值并发连接数
每用户日均操作数        →    峰值 TPS
单次操作数据量          →    网络带宽 / 存储增长
业务增长率             →    资源需求增长曲线
```

计算示例：

```
电商场景:
  DAU = 100,000
  峰值倍率 = 5x（大促）
  每用户日均 50 次请求
  高峰集中在 4 小时内

  日总请求 = 100,000 × 50 = 5,000,000
  高峰 TPS = 5,000,000 / (4 × 3600) × 5 = 1,736 req/s

  Little's Law (avg response 100ms, think time 2s):
  并发用户 = 1,736 × (0.1 + 2) = 3,646
```

**Step 2: 资源建模 — 从技术指标到硬件需求**

单机容量测试（stress test）确定：

```
单机极限:
  CPU 80% 时 TPS = 500 req/s
  内存: 4GB 足够（无泄漏）
  网络: 100Mbps 未打满

需求: 1,736 req/s
所需实例 = 1736 / 500 = 3.47 → 4 台（不含冗余）
+ 冗余 (N+1): 5 台
+ 容灾 (跨 AZ): 6 台
```

**Step 3: 验证与安全系数**

```
安全系数 = 实际部署容量 / 峰值需求

| 安全系数 | 含义 | 适用场景 |
|----------|------|---------|
| 1.0x | 刚好够用 | 不可接受 |
| 1.3x | 30% 余量 | 可预测的内部系统 |
| 1.5x | 50% 余量 | 一般互联网应用 |
| 2.0x | 100% 余量 | 金融、支付等关键系统 |
| 3.0x+ | 200%+ 余量 | 大促、秒杀场景 |
```

### 9.3 容量模型类型

| 模型                                | 方法                         | 优点           | 缺点             |
| ----------------------------------- | ---------------------------- | -------------- | ---------------- |
| **线性外推**                        | 当前 TPS × 增长倍率          | 简单直观       | 忽略非线性瓶颈   |
| **USL (Universal Scalability Law)** | 考虑串行化 + 协调开销        | 预测扩展性拐点 | 需要多组实测数据 |
| **排队论**                          | M/M/c 模型，精确计算等待时间 | 数学严谨       | 假设条件严格     |

### USL 模型详解

```
C(N) = N / (1 + α(N-1) + β·N(N-1))

N = 并发数
α = 串行化系数（排队、锁竞争）
β = 协调系数（缓存一致性、分布式协调）
C(N) = 相对吞吐量
```

```
                  C(N)
吞吐量              │        ╭──── 理想线性 (α=0, β=0)
                   │      ╭─┤
                   │    ╭─┤ ╰── Amdahl 瓶颈 (β=0)
                   │  ╭─┤ ╰────
                   │╭─┤ ╰────── USL 回退 (β>0)
                   ├──┴─────────────────
                   └──────────────────── N (并发数)

β > 0 时，超过拐点后加机器反而更慢（协调开销超过处理能力）
```

实际应用：

```
3 次 stress test 数据:
  50 threads  → 48 TPS
  100 threads → 90 TPS  (理想应为 96)
  200 threads → 138 TPS (理想应为 192)

效率递减: 96% → 90% → 69%
→ α ≈ 0.004, β ≈ 0.00003
→ 理论最大吞吐量拐点 ≈ 183 threads
→ 超过 183 threads 加并发不增加吞吐量
```

### 9.4 容量规划报告模板

```
1. 业务预测
   - 当前: DAU 10万, 峰值 TPS 200
   - 6个月后: DAU 30万, 峰值 TPS 600 (预估)
   - 12个月后: DAU 80万, 峰值 TPS 1600 (预估)

2. 单机容量基线 (stress test 结果)
   - CPU 80% 时: 500 TPS, p95 = 120ms
   - 内存: 稳态 2.1GB, 无泄漏
   - 磁盘 I/O: 未饱和

3. 扩展性验证 (USL 建模)
   - 线性扩展范围: 1-4 实例 (效率 > 90%)
   - 拐点: 8 实例后效率快速下降 (DB 锁竞争)
   - 建议: 超过 6 实例需做 DB 分片

4. 资源规划
   | 时间点 | 需求 TPS | 实例数 | 安全系数 |
   |--------|---------|--------|---------|
   | 当前 | 200 | 2 | 5.0x |
   | 6个月 | 600 | 3 | 2.5x |
   | 12个月 | 1600 | 5+分片 | 1.6x |

5. 风险与建议
   - 12个月时安全系数降至 1.6x，需提前 3 个月启动 DB 分片
   - 建议每季度重跑 stress test 更新基线
```

## 10. 性能调优系统化思路

### 10.1 调优原则

**先测量，再优化。不要猜。**

```
测量 → 定位瓶颈 → 优化 → 验证 → 下一个瓶颈
  ↑                              │
  └──────────────────────────────┘
```

三条铁律：

1. **不要过早优化** — 先证明有性能问题
2. **不要同时优化多个点** — 无法确认哪个生效
3. **优化后必须回归** — 防止优化引入功能 bug

### 10.2 分层排查法

性能问题必然出在某一层。从上到下逐层排查：

```
┌─────────────────────────────────┐
│  Layer 1: 网络层                  │  DNS, TCP, TLS, 带宽, CDN
├─────────────────────────────────┤
│  Layer 2: 接入层                  │  Nginx/LB 连接数, keepalive, 限流
├─────────────────────────────────┤
│  Layer 3: 应用层                  │  线程池, 连接池, 内存, GC, 锁
├─────────────────────────────────┤
│  Layer 4: 数据层                  │  SQL 慢查询, 索引, 连接池, 主从延迟
├─────────────────────────────────┤
│  Layer 5: 基础设施                │  CPU, 内存, 磁盘 I/O, 网卡
└─────────────────────────────────┘

排查顺序: 5 → 4 → 3 → 2 → 1
（先排除硬件瓶颈，再看软件）
```

### 10.3 每层的排查工具和指标

**Layer 5: 基础设施**

| 指标          | 健康标准        | 排查工具            | 超标含义               |
| ------------- | --------------- | ------------------- | ---------------------- |
| CPU 使用率    | < 70% (持续)    | top, vmstat, sar    | 计算瓶颈或 GC 过频     |
| 内存使用率    | < 80%           | free, /proc/meminfo | 可能触发 swap 或 OOM   |
| 磁盘 I/O wait | < 20%           | iostat, iotop       | 日志写入或 DB 磁盘瓶颈 |
| 网络带宽      | < 70%           | iftop, nethogs      | 响应体过大或日志传输   |
| TCP 连接数    | < ulimit 的 80% | ss, netstat         | 连接泄漏或未复用       |

**Layer 4: 数据层**

| 问题       | 排查方法                  | 常见优化               |
| ---------- | ------------------------- | ---------------------- |
| 慢查询     | slow query log, EXPLAIN   | 加索引、优化 SQL       |
| 连接池耗尽 | SHOW PROCESSLIST          | 增大连接池、优化长事务 |
| 锁等待     | SHOW ENGINE INNODB STATUS | 减小事务粒度、乐观锁   |
| 主从延迟   | SHOW SLAVE STATUS         | 优化大事务、并行复制   |
| 全表扫描   | EXPLAIN type=ALL          | 加索引                 |

**Layer 3: 应用层**

| 问题     | 排查方法                      | 常见优化                 |
| -------- | ----------------------------- | ------------------------ |
| 线程池满 | 线程 dump (jstack)            | 增大线程池、异步化       |
| 连接池满 | 连接池监控 (HikariCP metrics) | 增大池大小、减少持有时间 |
| GC 停顿  | GC log (-Xlog:gc\*)           | 调整堆大小、换 GC 算法   |
| 内存泄漏 | heap dump (jmap) + MAT 分析   | 修复泄漏源               |
| 锁竞争   | 线程 dump 看 BLOCKED          | 减小同步范围、CAS 替代   |
| N+1 查询 | APM trace / SQL log           | 批量查询、JOIN           |

**Layer 2: 接入层**

| 问题                | 排查方法                                  | 常见优化                |
| ------------------- | ----------------------------------------- | ----------------------- |
| Nginx worker 连接满 | nginx error log                           | worker_connections 调大 |
| upstream 超时       | nginx access log (upstream_response_time) | 调整 proxy_read_timeout |
| 负载不均            | 各实例 TPS 对比                           | 换负载均衡算法          |
| keepalive 未启用    | 抓包看 TCP 握手频率                       | 启用 upstream keepalive |

**Layer 1: 网络层**

| 问题       | 排查方法         | 常见优化              |
| ---------- | ---------------- | --------------------- |
| DNS 解析慢 | dig, nslookup    | 本地缓存、/etc/hosts  |
| TLS 握手慢 | openssl s_client | Session 复用、TLS 1.3 |
| 高延迟     | ping, traceroute | CDN、就近部署         |
| 丢包       | mtr, tcpdump     | 检查网卡/交换机       |

### 10.4 调优优先级矩阵

**不是所有优化都值得做。** 评估两个维度：

```
            收益高
              │
  ① 必须做    │  ② 优先做
  (高收益低成本) │  (高收益高成本)
              │
──────────────┼──────────────
              │
  ③ 不急      │  ④ 不做
  (低收益低成本) │  (低收益高成本)
              │
            收益低
       成本低 ←──→ 成本高
```

常见优化的分类：

| 象限     | 优化手段                       | 示例                 |
| -------- | ------------------------------ | -------------------- |
| ① 必须做 | 加索引、连接池复用、启用 gzip  | 改一行配置，TPS 翻倍 |
| ② 优先做 | SQL 重写、缓存层、读写分离     | 需要开发，但收益明确 |
| ③ 不急   | JVM 调参、Nginx 微调           | 提升 5-10%，有空再做 |
| ④ 不做   | 重写框架、换语言、过度微服务化 | 成本巨大，收益不确定 |

### 10.5 调优案例 — 系统化分析过程

```
现象: load test 50 threads, TPS 从 200 逐渐降到 50, p99 从 100ms 升到 3s

Step 1: 基础设施 (Layer 5)
  CPU: 45% → 排除
  内存: 3.2GB → 3.8GB (持续增长) → 可疑
  磁盘: 正常
  结论: 内存持续增长，疑似泄漏

Step 2: 数据层 (Layer 4)
  慢查询: 无
  连接池: 20/20 已满 → 可疑
  结论: 连接池满，新请求排队等待

Step 3: 应用层 (Layer 3)
  线程 dump: 30 个线程 WAITING on ConnectionPool.getConnection()
  连接持有时间: 从 10ms 增长到 500ms
  根因: 某个事务未 commit，连接泄漏

Step 4: 修复
  代码: try-with-resources 确保连接释放
  验证: 重跑 load test, TPS 稳定 200, 内存稳定 3.2GB

Step 5: 回归
  全部 unit tests PASS
  smoke/load/stress 结果正常
```

### 10.6 Node.js / Express 特有的性能考量

| 问题         | 原因                               | 排查                          | 优化                        |
| ------------ | ---------------------------------- | ----------------------------- | --------------------------- |
| 事件循环阻塞 | 同步操作（CPU 密集计算、同步 I/O） | `--prof` + tick processor     | Worker Threads、异步化      |
| 内存泄漏     | 闭包引用、全局缓存无上限           | `--inspect` + Chrome DevTools | WeakRef、LRU 缓存           |
| 连接数限制   | 默认 maxSockets                    | `http.globalAgent.maxSockets` | 调大或禁用 Agent            |
| SQLite 锁    | WAL 模式下仍有写锁                 | better-sqlite3 是同步的       | 写操作排队、或换 PostgreSQL |

## 11. 分布式压测架构

### 11.1 为什么需要分布式

单机压测的上限：

| 工具   | 单机上限     | 瓶颈                        |
| ------ | ------------ | --------------------------- |
| JMeter | ~500 threads | JVM 内存 + 线程上下文切换   |
| k6     | ~10,000 VUs  | Go 协程，CPU 和网络先到瓶颈 |

超过单机上限时需要分布式：

```
┌──────────────┐
│  Controller   │  分发测试计划、汇总结果
└──────┬───────┘
       │
  ┌────┼────┬────────┐
  ▼    ▼    ▼        ▼
┌───┐┌───┐┌───┐   ┌───┐
│ W1 ││ W2 ││ W3 │...│ Wn │  Worker 节点，执行实际压测
└───┘└───┘└───┘   └───┘
```

### 11.2 JMeter 分布式模式

```bash
# Worker 节点 (每台机器):
jmeter-server -Djava.rmi.server.hostname=<worker-ip>

# Controller 节点:
jmeter -n -t test.jmx -R worker1-ip,worker2-ip,worker3-ip -l results.jtl
```

| 注意事项     | 说明                                         |
| ------------ | -------------------------------------------- |
| threads 分配 | 100 threads + 3 workers = 每台 33-34 threads |
| 数据文件     | CSV 等参数文件需同步到所有 worker            |
| 网络         | Controller 和 Worker 之间需低延迟            |
| 时钟同步     | NTP 同步，否则聚合结果时间线错乱             |
| 结果汇总     | Controller 自动合并所有 Worker 的 .jtl       |

### 11.3 k6 分布式方案

k6 本身不内置分布式，常见方案：

| 方案                         | 复杂度 | 适用场景       |
| ---------------------------- | ------ | -------------- |
| k6 Cloud (Grafana Cloud)     | 低     | SaaS，按需付费 |
| k6-operator (Kubernetes)     | 中     | 已有 K8S 集群  |
| 多机手动运行 + InfluxDB 汇总 | 高     | 完全自控       |

```yaml
# k6-operator on Kubernetes
apiVersion: k6.io/v1alpha1
kind: K6
metadata:
  name: load-test
spec:
  parallelism: 4 # 4 个 Pod 并行
  script:
    configMap:
      name: load-test-script
```

### 11.4 何时需要分布式

| 场景               | 是否需要 | 理由                       |
| ------------------ | -------- | -------------------------- |
| < 200 threads      | 不需要   | 单机 JMeter 足够           |
| 200-1000 threads   | 视情况   | JMeter 需要，k6 单机可能够 |
| > 1000 threads     | 需要     | 两种工具都需要             |
| 压测机和目标同网段 | 不需要   | 网络不是瓶颈               |
| 跨区域压测         | 需要     | 模拟不同地理位置的用户     |

## 12. JMeter vs k6 — 双引擎对比

> 章节 9-11 为新增深度内容，以下为原有章节（编号顺延）

| 维度    | k6                        | JMeter                           |
| ----- | ------------------------- | -------------------------------- |
| 语言    | JavaScript                | XML (.jmx) + GUI                 |
| 执行模式  | CLI, 轻量                   | CLI (-n) 或 GUI                   |
| 资源消耗  | Go 协程，单机可支撑更高并发           | Java 线程，200+ threads 需调 JVM      |
| 报告    | 终端输出 + InfluxDB + Grafana | HTML Dashboard + .jtl + InfluxDB |
| CI 集成 | 原生 CLI，零依赖                | 需安装 JRE + JMeter                 |
| 企业接受度 | 新兴，DevOps 团队偏好            | 传统企业标准，PMO 认可度高                  |
| 适用场景  | 开发者自测、CI 门禁               | 正式性能报告、客户交付                      |

**为什么需要双引擎？**

- k6: CI 快速门禁（轻量、秒级启动）
- JMeter: 正式性能报告（HTML Dashboard、企业认可）

## 13. JMeter HTML 报告注意事项

| 问题              | 原因                         | 缓解                                    |
| ----------------- | ---------------------------- | --------------------------------------- |
| 开始/结束时间相同 | 时间戳精度为分钟级           | duration ≥ 60s                          |
| 图表数据太少      | 样本量不足                   | threads × duration 确保 ≥ 50 样本       |
| 吞吐量偏低        | ramp-up 阶段拉低平均值       | 关注满载期间数据                        |
| Connect Time 含义 | TCP 握手耗时（HTTPS 含 TLS） | localhost 测试通常 ≤ 1ms                |
| Latency 含义      | TTFB（首字节延迟）           | 含 Connect + 服务器处理，不含响应体传输 |
